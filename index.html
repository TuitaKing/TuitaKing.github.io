消息队列

在计算机科学中，消息队列（英语：Message queue）是一种进程间通信或同一进程的不同线程间的通信方式，软件的贮列用来处理一系列的输入，通常是来自用户。消息队列提供了异步的通信协议，每一个贮列中的纪录包含详细说明的资料，包含发生的时间，输入设备的种类，以及特定的输入参数，也就是说：消息的发送者和接收者不需要同时与消息队列交互。消息会保存在队列中，直到接收者取回它。 来自wiki。

同一进程的不同线程间通信使用比较多的就是线程池了，在jdk里面，线程池就是将runnable包装为一个个worker，然后放在队列中，由各种调度方式去调用worker。

而跨进程间通信使用的比较多的就是kafka，rabbitmq，rocketmq，activemq等等消息队列中间件。

canal 框架和代码结构

canal主要用于解析binlog日志，主要用于数据库实时备份和数据库数据同步。之所以现在想写这个东西源码解析，主要是最近的工作需要用到，打算做一个伪实时数仓，其实就是mysql的数据同步到es中。由于canal第一次接触，所以打算看下它的源码，做个笔记，方便以后学习和遇到问题排查问题。canal的github地址。本文主要学习来源于田守枝

代码结构

上面主要模版的功能为：

common模块：主要是提供了一些公共的工具类和接口。

client模块：canal的客户端。核心接口为CanalConnector

example模块：提供client模块使用案例。

protocol模块：client和server模块之间的通信协议

parser模块：数据源接入，模拟slave协议和master进行交互，协议解析。parser模块依赖于dbsync、driver模块。

sink模块：parser和store链接器，进行数据过滤，加工，分发的工作。核心接口为CanalEventSink

store模块：数据存储。核心接口为CanalEventStore

meta模块：增量订阅&消费信息管理器，核心接口为CanalMetaManager，主要用于记录canal消费到的mysql binlog的位置，

记录一次logback锁造成生产环境响应速度慢的问题

Posted on 2020-03-23 Edited on 2020-11-24

问题

业务会在每天凌晨进行高强度运算，CPU使用率一直在90%左右。该业务比较特殊，日志量特别大，引入的新组件也会打日志。原本1个小时跑完的业务（大约170w左右的数据量），引入新组件后发现跑完需要3小时。

排查过程

该业务主要是消费mq，然后根据mq消息进行rpc调用，查看rpc调用日志发现每次rpc调用时间并没有很长，且服务段和消费端的线程池没有满且基本没有被使用（200 core 仅仅不到10 被使用，30s采样一次），初步确定和rpc性能没有关系。晚上发现该业务的cpu使用率由原本的90下降到几乎只有40，查看mq的provider也没有发现速度变慢，mq的消息被消费的十分的均匀，感觉是限流或者锁。在线上通过jstack配合top -Hp、jstate、jmap等发现并没有gc或者其他的异常，但是发现基本上所有的业务线程都处于WAITING状态，且waiting位置在logback的subAppend上。然后根据情况Google之。查出结果https://jira.qos.ch/browse/LOGBACK-1378,后升级发现问题解决。

java线程池运行原理

Posted on 2020-03-12 Edited on 2020-11-24

卷首语

本文尝试对java线程池的运行进行一个源码的分析。主要是创建的过程，线程运行的过程等。

首先上总结，便于记忆和后面长长的代码懒得看的时候能够回想起来。

  • 线程池主要靠一个原子AtomicInteger ctl 来表示线程池运行状态，其中前三位标示为状态，后29位标示线程池中运行的线程数。
  • 线程池的策略是，小于核心线程数则新建线程，大于核心线程数且任务队列没满则添加任务到队列中，队列和核心线程数都满了的时候，如果没有超过最大线程数，则继续新建线程。如果三者都满了，则拒绝服务。
  • 线程池内部将runnable封装成worker,worker也实现了Runnable，在线程池执行中传入的runnable其实是没有直接和线程绑定的，而是通过在worker的线程执行的时候被调用的。
  • 当前没有任务执行的时候，当前的线程是阻塞在从任务队列中获取任务的。
  • 如果执行了shutdown则线程会将任务队列中的任务继续执行，但是不添加新的任务。关闭线程是将worker中的线程全部执行interrupt，这里执行的是interruptIdleWorkers。
  • 如果执行了shutdownnow则不会等待任务执行，直接关闭线程，调用的是interruptWorkers

一次nginx调优过程

Posted on 2020-02-26 Edited on 2020-11-24

问题

nginx作为代理，接受pinpoint-agent的9994,9995,9996的tcp和udp请求到pinpoint-collector集群（4台服务器）。每天到零点的时候数据量飙升，高并发情况下，pinpoint的链路数据丢失严重。

排查过程

首先因为公司网络环境复杂，数据在从nginx到collector可能经过n>3层网络转发。初期一位是collector的集群机器太少，后增加到6台。数据量仍然丢失，后查看nginx系统负载，系统负载很小，不会影响到性能，collector负载也很小，gc日志和应用日志也正常，系统瓶颈没有在collector到hbase。

最后在0点登录nginx系统，直接被踢出和登录不上去的情况。但是系统负载此时也是正常的。在被踢出前发现ksoftirqd进程cpu飙升到100%。查看nginx的error日志发现大量的101错误。

查看top发现大量的nginx-worker都在使用软中断的CPU

软中断过高应该是该系统的瓶颈，网络上查询到软中断过高主要是由于调用系统服务。怀疑是nginx造成的。根据nginx官方文档，发现有个reuseport被加在每一个udp模块中。感觉找到了方向。加上后，数据果然恢复正常。

关于reuseport

该配置是linux3.9以后的新特性，可以让多个进程监听同一个端口。这么做的好处在于减少了多个worker进程对accept方法的争夺。在内核态进行socket的负载均衡。

关于multi_accept

该配置（上文提到的nginx这个配置是打开的）的主要作用是用于单核能够更加多的接收链接。存在一个忙，其他都看着的情况。

Codis 启动总览

Posted on 2020-02-20 Edited on 2020-11-24

卷首语

本文主要是来查看codis每一次部署和启动的时候做了什么事情。

  • Codis Server：基于 redis-3.2.8 分支开发。增加了额外的数据结构，以支持 slot 有关的操作以及数据迁移指令。具体的修改可以参考文档 redis 的修改。

  • Codis Proxy：客户端连接的 Redis 代理服务, 实现了 Redis 协议。 除部分命令不支持以外(不支持的命令列表)，表现的和原生的 Redis 没有区别（就像 Twemproxy）。

    • 对于同一个业务集群而言，可以同时部署多个 codis-proxy 实例；
    • 不同 codis-proxy 之间由 codis-dashboard 保证状态同步。

  • Codis Dashboard：集群管理工具，支持 codis-proxy、codis-server 的添加、删除，以及据迁移等操作。在集群状态发生改变时，codis-dashboard 维护集群下所有 codis-proxy 的状态的一致性。

    • 对于同一个业务集群而言，同一个时刻 codis-dashboard 只能有 0个或者1个；
    • 所有对集群的修改都必须通过 codis-dashboard 完成。

  • Codis Admin：集群管理的命令行工具。

    • 可用于控制 codis-proxy、codis-dashboard 状态以及访问外部存储。

  • Codis FE：集群管理界面。

    • 多个集群实例共享可以共享同一个前端展示页面；
    • 通过配置文件管理后端 codis-dashboard 列表，配置文件可自动更新。

  • Storage：为集群状态提供外部存储。

    • 提供 Namespace 概念，不同集群的会按照不同 product name 进行组织；
    • 目前仅提供了 Zookeeper、Etcd、Fs 三种实现，但是提供了抽象的 interface 可自行扩展。
